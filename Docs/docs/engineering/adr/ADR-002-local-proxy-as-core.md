# ADR-002: 采用本地统一代理作为底层内核

- Status: Accepted
- Date: 2026-04-05

## Context

PCE 希望覆盖用户已经在使用的多种 AI 工具，而不是要求用户迁移到一个新壳里重新开始。

如果只走单个平台插件、单个 IDE 扩展或单个协议，会很快遇到以下问题：

- 覆盖范围碎片化
- 无法统一记录多来源数据
- 用户需要理解并配置多套接入方式
- 未来做注入时还得重建新的底层能力

## Decision

PCE 采用“本地统一代理 / AI 网关”作为核心基础设施。

其职责是：

- 处于用户与上游 AI 服务之间
- 统一承接 API 工具、IDE 插件、CLI、实验项目等交互流量
- 记录请求与响应
- 将不同来源归一化为统一本地结构
- 为未来注入式干预预留同一条数据通道

## Rationale

1. 代理是最符合“一次接入、多处受益”的底层形态。
2. 它既适合当前只读记录，也天然适合未来读写注入。
3. 它把“接入问题”从单工具适配，转成一条长期可复用的基础设施。
4. 它更接近“统一 AI 网关”的产品心智，而不是附着在某个平台上的外挂。

## Consequences

### Positive
- 形成统一的数据入口
- 未来注入不需要重建基础设施
- 多工具、多来源的数据第一次可以落到一套本地结构里
- 技术路线和产品心智更一致

### Negative
- 需要处理系统代理、证书、allowlist 等底层问题
- 对隐私、脱敏、稳定性要求更高
- 完整模式会引入更重的安装与授权体验

## Guardrails

为了让这条路线可控，必须同时遵守：

- allowlist 只拦截目标 AI 域名
- 默认本地存储
- 敏感头与认证信息必须脱敏
- 失败时优先保证用户正常调用
- 第一阶段只读不写
