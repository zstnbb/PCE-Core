# ADR-001: 第一阶段先做记录，不直接做干预

- Status: Accepted
- Date: 2026-04-05

## Context

PCE 的原始主张天然会把人推向“尽快做 infer、尽快做帮助姿态调节、尽快做注入”。

但在当前阶段，下面这些核心问题都还没有被可靠解决：

- infer 引擎的机制与可信度
- 从原始交互中自动提取稳定信号的质量
- 冷启动时系统到底能提供什么价值
- 干预效果如何验证

如果在这些都不稳定的情况下先做干预，系统会过早建立在推测之上，而不是建立在足够扎实的事实之上。

## Decision

第一阶段的产品路径明确为：

`记录 -> 看见 -> 理解 -> 干预`

第一阶段只实现：

- 原始交互记录
- 本地持久化
- 统一查询与可视化回看

第一阶段明确不实现：

- 自动 infer
- 自动连续性建模
- 注入式帮助姿态调节

## Rationale

1. 没有稳定底层，就没有可信的上层干预。
2. 原始记录本身就是成立的用户价值。
3. 先看到自己的认知活动，再谈如何改变它，路径更符合项目自身主张。
4. 只有先积累数据，后续“理解 / 干预”才可能被验证。

## Consequences

### Positive
- 第一版目标更聚焦
- 技术风险显著下降
- 能更快验证“统一记录 + 本地可查”是否成立
- 后续架构会更稳，不容易反复推翻

### Negative
- 第一版不会立刻呈现最性感的“自动调校 AI”效果
- 需要克制，不把高级功能塞回当前周期

## Review Trigger

当下面条件同时出现时，可考虑进入“理解 / 干预”阶段：

- 已经形成足够稳定的原始数据底座
- 用户能明确感受到“看见”本身的价值
- 会话归并、搜索与回看已可用
- 至少有一套初步连续性结构能被人工验证
