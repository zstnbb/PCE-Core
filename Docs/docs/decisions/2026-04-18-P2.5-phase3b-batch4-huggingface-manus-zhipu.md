## Summary

Fourth round of the site-extractor migration. Ports `huggingface.js`,
`manus.js`, and `zhipu.js` to TypeScript on top of the capture-runtime
factory. No new runtime features in this batch — all three extractors
fit cleanly into the patterns established in batches 1-3.

**12 of 13 site extractors now on TypeScript.** Only `generic.js`
(serving Mistral + Kimi) remains, plus the three site-independent
legacy helpers (`detector.js` / `behavior_tracker.js` /
`text_collector.js`). Batch 5 finishes the migration.

## Design Decisions

### Why Zhipu uses `requireBothRoles: true` despite legacy's asymmetric guard

The legacy `zhipu.js` guard was:

```js
if (
  allMessages.some((msg) => msg.role === "user") &&
  !allMessages.some((msg) => msg.role === "assistant")
) {
  scheduleCapture(1500);
  return;
}
```

This defers capture when user messages exist but no assistant yet. It
does NOT defer the symmetric case (assistant-only, no user). The shared
`requireBothRoles: true` option defers both cases.

**Decision**: use the symmetric guard anyway. The assistant-only case
is practically impossible on Z.ai — conversations always start with
a user prompt. The symmetric guard is marginally more defensive (it
would catch a scenario where only an assistant message was extracted
due to a DOM-timing race where the user-side element hadn't yet
mounted). No behaviour drift observable in practice; zero additional
runtime complexity.

### Why the runtime now captures `message_update` payloads that the legacy Zhipu JS dropped

The legacy `captureConversation` early-exited when `newMessages.length
=== 0`:

```js
const newMessages = allMessages.slice(sentCount);
if (newMessages.length === 0) {
  lastFingerprint = fp;
  return;
}
```

This happens when `sentCount === allMessages.length` but the
fingerprint changed — which means the content of an already-sent
message changed (e.g. the last assistant message expanded during
streaming). The legacy JS silently dropped these updates.

The runtime's incremental-mode `message_update` branch instead sends
the last message with `capture_mode: "message_update"`. This IS a
behaviour change vs. legacy Zhipu, but:

1. It's a strict upgrade: streaming updates are now captured instead
   of silently dropped.
2. Backend dedup (`pipeline._message_hash` + session-hint rollup in
   `server.py`) already handles duplicate sends — net effect on the
   database is the same captured data + richer streaming coverage.
3. Matches the behaviour of every OTHER extractor in the shared
   runtime, including the 9 already on TypeScript.

Preserving the legacy drop would require adding a dedicated
`skipMessageUpdates?: boolean` runtime option with no other callers.
Not worth the API surface.

### Why Manus keeps `"Manus"` as the ultimate model-name fallback

Legacy parity. `manus.js` explicitly returned `"Manus"` when no DOM
element yielded a model name. Other TS extractors return null in the
same scenario. Neither is wrong — the difference is whether the
payload carries a generic label vs. no label. Python backend
normalises missing model_name to `"unknown"` anyway. The TS port
preserves the legacy convention.

### Why HuggingFace keeps the 3-stage model-name ladder with title-prefix fallback

The title-prefix heuristic only fires when the title prefix contains
a `/` (indicating `<org>/<model>` format). This is a deliberate
defence against matching unrelated title prefixes like `"Chat with
Mixtral"`. Preserving the behaviour byte-for-byte.

## Risks mitigated

- **Per-site regressions**: 45 new Vitest cases enumerate the
  extraction heuristics, role detection, noise stripping, and
  attachment handling for each site.
- **Zhipu deferred captures**: The `requireBothRoles` option covers
  the primary legacy guard (user-only). The documented intentional
  deviation on `message_update` is a net-positive behaviour change.
- **Manus attachment drift**: `extractLocalAttachments` is
  implemented + tested identically to the legacy code path.
  Same-origin skip protects against self-referential links.
- **HuggingFace title-prefix false positives**: The `/`-containing
  check preserves the legacy safeguard.
