## Summary

The final behavioural cleanup of the P2.5 migration. No new TS code —
this PR validates the Phase 3b migration by deleting the now-unused
legacy `content_scripts/` directory and rewiring
`entrypoints/background/injector.ts` to use the TS universal extractor
as its sole dynamic-injection payload.

After this PR, the dynamic-injection code path carries a single
constant (`UNIVERSAL_EXTRACTOR_TS_PATH = "universal-extractor.js"`)
instead of 19 per-domain script mappings. The `injector.ts` module
shrinks from ~325 lines to ~175 lines. All 39 legacy-table tests in
`injector.test.ts` are replaced with 9 cases covering the new
simplified API surface. 527 Python tests still pass.

## Deliverables

### 1. `entrypoints/background/injector.ts` rewrite (~50% smaller)

**Removed** (all were legacy-JS-path plumbing):

- `DOMAIN_CONTENT_SCRIPTS` — `Record<hostname, file-list>` covering 19

- `toContentScriptPaths(files)` — `"content_scripts/" + file` prefix.
- `getExtractorFlagForDomain(domain)` — table lookup for the probe flag.

**Kept** (new simplified API):

```ts
export const UNIVERSAL_EXTRACTOR_TS_PATH = "universal-extractor.js";

export function getDynamicScriptsForDomain(_domain: string): string[] {
  return [UNIVERSAL_EXTRACTOR_TS_PATH];
}

export function getScriptsForUrl(
  url: string,
  _opts: { includeDetector?: boolean } = {},
): string[] | null {
  // null for non-HTTP(S) / unparseable; else [UNIVERSAL_EXTRACTOR_TS_PATH]
}

export interface PipelineState {
  bridge: boolean;
  extractor: boolean;
  behavior: boolean;
}

export async function getTabPipelineState(tabId: number): Promise<PipelineState>
export async function tabHasExpectedPipeline(tabId: number): Promise<boolean>

export async function proactiveInjectFallback(ctx, tabId, url): Promise<void>
```

The public injection entry points (`handleDynamicInjection`,
`proactiveInjectFallback`) keep their signatures — `background.ts`
needed zero edits.

`getTabPipelineState` dropped its `domain` argument; the in-tab probe
now evaluates all 13 site-specific flags OR
`__PCE_UNIVERSAL_EXTRACTOR_LOADED` in a single `executeScript` closure.

## Behaviour Parity Checklist

| Pre-Phase-4 behaviour | Post-Phase-4 behaviour | Parity |
|---|---|---|
| Covered-site navigation → manifest content_scripts fire → site extractor loads | Covered-site navigation → WXT-registered `defineContentScript` matches → TS extractor loads | ✅ exact |
| Detector fires on unknown AI site → injector looks up `DOMAIN_CONTENT_SCRIPTS[domain]` OR `UNIVERSAL_FALLBACK_SCRIPTS` | Detector fires on unknown AI site → injector loads `universal-extractor.js` (single file) | ✅ richer (shared runtime inherits `requireBothRoles`, streaming deferral, rollback) |
| Proactive onUpdated fallback on covered site → injector re-runs the full per-domain pipeline | Proactive onUpdated fallback on covered site → injector loads `universal-extractor.js`, which bails out via `SITE_EXTRACTOR_FLAGS` guard when the site extractor is already active | ✅ no double-capture |
| `getTabPipelineState(tabId, domain)` checks the domain's specific flag | `getTabPipelineState(tabId)` checks all 13 site flags OR universal flag | ✅ functionally equivalent |
| `tabHasExpectedPipeline` depended on domain flag | No domain argument — checks generic pipeline state | ✅ looser but semantically correct |

## Design Decisions

### Why a single payload for both known + unknown AI sites?

Before Phase 4, the injector's dynamic-injection path carried the
full per-domain pipeline (6 helpers + 1 site extractor = 7 files).
That was necessary when extractors were separate JS files that
depended on shared helpers living at known filenames.

After Phase 3b, every TS extractor imports its dependencies via ESM.
The `universal-extractor.js` WXT output has everything inlined. And
critically: the universal extractor already has a
`SITE_EXTRACTOR_FLAGS` guard that bails out when a site-specific
extractor is active. So injecting it on a covered site is safe —
it'll self-disable.

One payload → two modes:
  - Covered site, static content_scripts already loaded → universal
    bails out via `SITE_EXTRACTOR_FLAGS`.
  - Unknown AI site OR covered site where static content_scripts
    didn't fire → universal drives the shared capture runtime.

### Why not move `DOMAIN_EXTRACTOR_FLAGS` into a shared constant?

The only remaining consumer was `getTabPipelineState`'s in-tab
closure, which now inlines the 13 flags directly. Moving them to a
shared module would require serialising them across the
`chrome.scripting.executeScript` boundary (the target function runs
in the tab's isolated world and can't reach into the service worker's
module graph). Inlining keeps the closure self-contained and
independently readable.

### Why keep `proactiveInjectFallback` alive?

It catches three real-world edge cases:

1. **Tab came back from discard**: Chrome discarded the tab under
   memory pressure; when the user revisits, the static
   content_scripts don't re-run.
2. **CDP / automation**: Puppeteer / Playwright / Selenium can
   navigate tabs without firing the normal content_scripts pipeline.
3. **SPA race with `run_at: document_start`**: On slow networks, the
   content script's `main()` can fire before React / Angular mounts
   the first interactive DOM. The observer then gets no mutations to
   react to, so the capture never triggers.

The proactive fallback gives us a second chance by running
`universal-extractor.js` 1.5 s after `chrome.tabs.onUpdated`. If the
static content script already ran, the universal extractor no-ops.
If not, we get a functional capture.

### Why keep `opts.includeDetector` on `getScriptsForUrl` as a no-op?

`background.ts` and any downstream code might call `getScriptsForUrl`
with the option set; breaking the signature would require rippled
edits. The cost of keeping the option as an accepted-but-ignored
parameter is ~2 lines of function-signature noise. Phase 4c's root
README update can revisit if a broader ABI cleanup is worth doing.

### Why not Phase 4 in one PR?

Three small PRs scale better for review:

- **Phase 4a (this PR)**: the logic surgery. Injector rewire,
  legacy dir deletion, sync script + gitignore narrowing.
- **Phase 4b**: asset relocation. Move `icons/` + `popup/` directly
  into `public/`, delete `sync-legacy-assets.mjs`, remove `prebuild`
  + `predev` hooks, simplify `.gitignore`.
- **Phase 4c**: documentation + cross-workspace installer.

Each follow-up is small and self-contained. Phase 4a is where the
logic risk lives, so it gets the most review attention.

## Risks mitigated

- **Dynamic-injection regression**: 9 new test cases assert the
  simplified `injector.ts` API. The behaviour parity table above
  enumerates every pre-Phase-4 code path and its post-Phase-4
  equivalent.
- **Accidental legacy-JS reference**: `grep`-verified that no
  production code references the deleted `content_scripts/*.js`
  paths (only docstring mentions remain).
- **Asset sync breakage**: `sync-legacy-assets.mjs` continues to copy
  `icons/` + `popup/`; WXT's manifest generator still resolves
  `popup/popup.html` + `icons/icon*.png` against `public/`.
- **Test-count preservation**: The 39 dropped tests were all
  table-lookup invariants that no longer have tables to defend. Their
  replacement is 9 cases against the simpler API — smaller surface
  area, tighter coverage.
