## Summary

Purely a reference sweep. Phase 4b deleted `../pce_browser_extension/`,
which left several production-code + e2e-test files pointing at a
directory that no longer exists. Phase 4c rewires every runtime
consumer to the WXT build output at
`pce_browser_extension_wxt/.output/chrome-mv3/`, updates the
user-facing installation docs accordingly, and centralises the
resolution logic in `pce_app/managed_browser.py`.

No behaviour change for code that hadn't been updated yet — the same
files that previously loaded the legacy dir now load the built bundle.
All 527 Python tests still pass.

## Deliverables

### 1. Extension-path resolution in `pce_app/managed_browser.py`

Replaced the static `_EXTENSION_DIR` constant with a `_find_extension_dir()`
helper that honours (in order):

  1. `PCE_EXTENSION_DIR` env var — absolute or expandable path, for
     overrides in CI / custom builds.
  2. `pce_browser_extension_wxt/.output/chrome-mv3/` — primary.
  3. `pce_browser_extension_wxt/.output/firefox-mv3/` — fallback for
     Firefox devs.
  4. The Chrome output path as a default (so the downstream error
     message, when the directory is missing, points at the expected
     location).

The rest of the `ManagedBrowser` class didn't change — it still
reads `_EXTENSION_DIR` at module load time.

## Design Decisions

### Why resolve the extension dir at module-import time instead of per-launch?

`ManagedBrowser.__init__` accepts an `extension_dir` kwarg for
explicit overrides; the default (`_EXTENSION_DIR`) is evaluated
once at module import and cached. This mirrors the pre-Phase-4a
behaviour and keeps the launch path synchronous. The env-var check
happens inside `_find_extension_dir()` so it's honoured.

If the user rebuilds the extension between launches, the cached path
still points at the right directory (WXT overwrites the bundle in
place during `pnpm build`, not creating a new path). So caching is
safe.

### Why not add a build-output existence check with a nice error message?

Two reasons:
  1. The WXT output is expected to exist when `pnpm build` has been
     run. A missing directory indicates the user hasn't run the
     build step, which is the correct thing to surface.
  2. `ManagedBrowser.launch()` already logs `"Extension: <path>"`
     before launching Chrome. Chrome itself will emit an error in
     its console when `--load-extension=<nonexistent>` is passed.
     Adding a Python-side check would duplicate that signal.

Phase 4d's e2e work will verify this path end-to-end; if the error
experience is too cryptic we can add an earlier guard then.

### Why the `PCE_EXTENSION_DIR` env override?

CI environments often build the extension into a non-standard
location (e.g. a shared cache directory). Packaged releases (via
PyInstaller) also put the extension somewhere other than the source
tree. The env var provides an escape hatch without requiring code
changes.

### Why update `E2E_STATUS_2026-04-10.md` and `IMPLEMENTATION-ROADMAP.md` don't need updates?

Both are frozen historical snapshots. The roadmap describes the plan
as of when it was written; the E2E status is dated. Rewriting them
would erase useful archaeology. Forward-looking docs (README, BUILD,
MANUAL_VERIFICATION) are the only ones that needed to change.

## Risks mitigated

- **Legacy directory runtime failures**: All 5 e2e test helpers +
  `managed_browser.py` now resolve against the WXT build output.
  The legacy path is gone from production code.
- **User confusion over the install flow**: Three user-facing docs
  (root README, BUILD.md, MANUAL_VERIFICATION_CHECKLIST.md) walk the
  user through `pnpm build` + load-unpacked steps. The dashboard's
  service card reads out the same steps via its `setup_detail` API.
- **CI / packaging flexibility**: `PCE_EXTENSION_DIR` env override
  lets CI pipelines and PyInstaller builds point at a non-default
  location without code changes.
- **Test-suite stability**: 527 Python tests (unit + non-e2e)
  continue to pass. E2E tests touch real file paths, so Phase 4d
  will re-run them after `pnpm build`.
