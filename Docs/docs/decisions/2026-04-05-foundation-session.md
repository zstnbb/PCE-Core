# Foundation Session

- Date: 2026-04-05
- Topic: PCE 第一版基础设施形态、阶段目标与执行路径
- Status: Accepted

## 本次讨论要解决的问题

1. PCE 第一版应该直接做“干预”，还是先打基础。
2. PCE 应该怎样接入用户已经在使用的多种 AI 产品。
3. 技术实现如何和对用户呈现的产品形态区分开。
4. 如何在“足够彻底”与“风险可控”之间找到第一版方案。

## 演化链

本轮讨论的演化路径可以概括为：

1. 从原始主张出发，意识到 infer 引擎、自动信号提取、冷启动与验证都尚未定型。
2. 因此把第一版目标从“直接帮助姿态调节”收束为“记录 + 通讯层 + 本地持久化”。
3. 在接入形态上，先经历了 adapter / MCP / 浏览器插件等设想。
4. 最终收束到“本地统一代理 / AI 网关”作为最扎实的底层内核。
5. 同时确认同一条管道未来既能只读记录，也能在成熟后承载注入与干预。

## 本次最终结论

### 1. 第一版不做干预，先做“记录 -> 看见”

PCE 的产品路径明确为：

`记录 -> 看见 -> 理解 -> 干预`

第一版必须先把前两步做扎实，而不是一开始就急着做 infer、注入或帮助姿态调节。

### 2. 当前阶段定义为“打地基”

这一阶段的核心不是“让模型更聪明”，而是把下面三件事打通：

- 记录
- 通讯层
- 本地持久化

只有这条基础设施先成立，后续的连续性建模、帮助姿态、注入式干预才有稳定承载体。

### 3. 第一层用户价值是“看到正在思考的自己”

PCE 第一版最直接的价值，不是改变模型行为，而是把分散在 Claude、GPT、Cursor、CLI 与自建项目里的 AI 交互汇总到本地，形成统一时间线、统一搜索入口和统一回看界面。

### 4. 技术底层收束为“本地统一代理 / AI 网关”

PCE 不再被定义成单个平台适配器，而是被定义成用户与各类 AI 产品之间的一层本地基础设施：

- API 工具、CLI、IDE 插件、自己的实验项目，都可以通过本地代理被记录
- Web 场景在轻量模式下可通过插件补足，在完整模式下可通过系统级代理统一覆盖

### 5. 采用“双模式”策略

为了兼顾覆盖率和用户承受度，第一版确认保留两种安装模式：

- 轻量模式：环境变量 / API 路由 + 浏览器插件
- 完整模式：系统级 HTTPS 代理 + 本地 CA 证书

完整模式是长期更彻底的方案，轻量模式是低门槛、低风险的兼容方案。

### 6. 同一条通道必须同时支持未来升级

第一版虽然只做只读记录，但底层基础设施必须从一开始就支持未来升级：

- 现在：读请求、读响应、存本地
- 未来：在请求进入上游模型前插入本地结构化上下文

这样就不需要在“干预阶段”重建一套新的底层。

### 7. 第一开发起点已经明确

第一开发动作不是做壳，也不是做推理，而是先完成一个最小的 Proxy PoC：

- `mitmproxy` 负责拦截和转发
- `SQLite` 负责本地持久化
- 先支持最小范围内的 OpenAI / Anthropic 域名
- 先把请求与响应稳定地存下来

## 当前明确不做的事

下列事项不是当前阶段的目标：

- 自动 infer 引擎
- 自动连续性模型更新
- 大规模 web 端注入
- 商业合规包装
- 企业控制台或云端能力
- 复杂的多模型策略编排

## 待后续决策的问题

这些问题还没有完全定型，但已经知道会出现：

1. 会话边界如何定义与归并。
2. 原始抓取数据与归一化结构之间如何分层。
3. 完整模式下的证书安装与系统代理配置体验。
4. 轻量模式下浏览器插件覆盖哪些平台。
5. 桌面外壳最终选 Tauri 还是 Electron。
6. 将来注入时的策略开关、透明度与风险控制。
