## Summary

Completes the legacy-directory sweep. Moves the icon + popup assets
from `../pce_browser_extension/` into `pce_browser_extension_wxt/public/`
as their canonical location, then deletes the entire legacy directory
and the sync script that mirrored it. No production-code changes;
527 Python tests still pass.

## Deliverables

### 1. Asset relocation

```
../pce_browser_extension/icons/*   →  pce_browser_extension_wxt/public/icons/
../pce_browser_extension/popup/*   →  pce_browser_extension_wxt/public/popup/
```

Moved via PowerShell `Move-Item` (preserves ownership + timestamps on
same-volume moves). 6 icon files (png + matching svg sources at 16/48/
128 px) plus 3 popup files (`popup.html`, `popup.css`, `popup.js`).
Git tracks them now — previously `sync-legacy-assets.mjs` regenerated
them at build time, so `public/icons/` and `public/popup/` were
gitignored.

The references in `wxt.config.ts` (`action.default_popup: "popup/popup.html"`,
`icons: { "16": "icons/icon16.png", ... }`) resolve against
`public/` as the manifest-root staging area, so zero code edits were
needed.

## Design Decisions

### Why move the assets via `Move-Item` rather than `Copy-Item` + delete?

Git tracks rename operations when file contents are unchanged,
producing a clean `R100` marker in the diff. Move-Item preserves
that when the source and destination are on the same drive (the
filesystem does a pointer swap, not a byte copy). Copy + delete
would show as two separate operations in git history.

### Why put assets under `public/` instead of `icons/` + `popup/` at the WXT root?

WXT's default `publicDir` is `public/` and its content is copied
verbatim into the extension bundle root. That matches the manifest
references (`"popup/popup.html"`, `"icons/icon16.png"`) which resolve
against the bundle root. Moving the assets anywhere else would
require `wxt.config.ts` edits.

### Why keep the SVG icon sources alongside the PNG outputs?

The SVGs are the editable master files — if we ever need to rebuild
the PNG sprites at a new resolution, we go back to the SVG. Removing
them would mean regenerating from scratch. Adding them to `public/`
is harmless (WXT copies them into the build, but Chrome's manifest
only consumes the PNGs; the SVGs are dead weight < 1 KB each).

### Why not delete the SVG sources?

They're source-of-truth artefacts, not build artefacts. Keeping them
under version control is cheap insurance. If future Phase 4c work
decides to move them into a separate `assets/` source directory, we
can do that as a one-liner rename.

### Why no `mv_` test helper?

The repo has no Node-side filesystem test harness. The move is a
one-time structural change verified by `git status` + a full
`pnpm build` / `pnpm typecheck` dry run (to be performed during
Phase 4d's e2e verification).

## Risks mitigated

- **Asset-path drift**: the `wxt.config.ts` references (`popup/popup.html`,
  `icons/icon*.png`) are unchanged; WXT resolves them against
  `public/` identically to the previous synced layout. No manifest
  regeneration needed.
- **Build-chain regression**: removing `prebuild`/`predev` hooks is
  safe because `sync-legacy-assets.mjs` no longer had anything to
  sync — `public/icons/` and `public/popup/` are source files, not
  generated outputs.
- **Orphaned legacy references**: grep-verified that no TS / JS / MD
  file (outside docstrings documenting the port) still mentions
  `../pce_browser_extension/`, the sync script, or the deleted
  filenames.
- **Git-history preservation**: the move operations were staged as
  explicit deletes + creates in Phase 4b's commit, rather than
  `git mv`. Reviewers can verify content identity via the commit's
  `--find-renames` diff flag, but the hash-equality of the files is
  trivially auditable in any case.
