## B. Why port shared helpers now instead of Phase 3b

The 13 site extractors (landing in Phase 3b‚Äì3f as 5 PRs of 2‚Äì3 each)
all depend on the same three helpers:

- `pce_dom_utils.js` ‚Äî generic attachment/reply extraction.
- `selector_engine.js` ‚Äî resilient CSS selector resolution.
- `site_configs.js` ‚Äî per-site selector tables.

If each Phase 3b PR re-imported from legacy JS globals
(`window.__PCE_EXTRACT`, `window.__PCE_SELECTOR_ENGINE`,
`window.__PCE_SITE_CONFIGS`), the TS ports would inherit a brittle
runtime-global coupling that defeats half the point of the migration.

Doing the shared ports up-front means every Phase 3b site extractor
can simply `import { createSelectorEngine } from "@/utils/selector-engine"`
and get typed, unit-tested helpers with IntelliSense. The legacy
globals still exist (fed by the still-shipped legacy JS), but TS site
extractors will bypass them in favour of direct imports.

## C. Shared-helper port design decisions

### `utils/site-configs.ts`

Exported shape:

```ts
interface PceSiteConfig {
  provider: string;
  sourceName: string;
  container: string[];
  userMsg: string[];
  assistantMsg: string[];
  turnPair: string[];
  streamingIndicator: string[];
  roleAttr: string[];
  roleKeywords: { user: string[]; assistant: string[] };
  sessionHintPattern: RegExp | null;
  modelSelector: string[];
}

const PCE_SITE_CONFIGS: Readonly<PceSiteConfigs>;  // frozen
function resolveSiteConfig(hostname: string): PceSiteConfig | null;
```

Aliases (`chat.openai.com`, `www.kimi.com`, `kimi.moonshot.cn`,
`chatglm.cn`, `chat.zhipuai.cn`) reuse the same config object by
reference (not a copy) ‚Äî same semantics as the legacy JS's
`PCE_SITE_CONFIGS["chat.openai.com"] = PCE_SITE_CONFIGS["chatgpt.com"]`.
The tests lock that in (`toBe`, not `toEqual`).

### `utils/selector-engine.ts`

Factory function instead of a singleton IIFE:

```ts
function createSelectorEngine(options?: SelectorEngineOptions): SelectorEngine;

interface SelectorEngineOptions {
  win?: Window;
  doc?: Document;
  hostname?: string;    // overrides location.hostname
  pathname?: string;    // overrides location.pathname
  cacheMissThreshold?: number;
  logger?: (...args: unknown[]) => void;
}
```

The legacy IIFE hard-coded `document` and `location` ‚Äî fine in a real
content-script context, impossible to test. The factory accepts
injectable handles that default to the real globals. Tests pass
happy-dom `document` + a fake hostname/pathname.

Preserved behaviours:

- 3-strikes-then-invalidate cache: `CACHE_MISS_THRESHOLD = 3`.
- Fast path: try cached index first, on miss scan all candidates.
- `detectRole` 3-strategy ladder: role attrs ‚Üí class keyword ‚Üí parent
  class keyword.
- `queryAllMessages` fallback: combined CSS selector ‚Üí turn-pair.
- `getDiagnostics` shape identical to legacy (hostname, configFound,
  provider, per-category cache state + selector hit counts).

Added: `resetCache()` for tests.

### `utils/pce-dom.ts`

All legacy helpers preserved:

- `normalizeCitationUrl(url, base?)` ‚Äî adds optional `base` param for
  tests; defaults to `location.href` in browser.
- `fingerprintConversation(messages)` ‚Äî same algorithm byte-for-byte.
- `inferMediaTypeFromName(name)` ‚Äî same 22-entry MIME map.
- `extractAttachments(el)` ‚Äî same 7-phase pipeline: images ‚Üí code
  blocks ‚Üí file chips ‚Üí heuristic file-name chips ‚Üí text-fallback
  file name ‚Üí citations ‚Üí canvas ‚Üí tool calls ‚Üí tool output.
- `extractThinking(el)` ‚Äî details ‚Üí class-keyword fallback.
- `extractReplyContent(el)` ‚Äî selector-priority ‚Üí clone-and-strip
  fallback.
- `isStreaming(doc?)` ‚Äî injectable doc; same selector lists.
- `getSessionHint(pathname?)` ‚Äî injectable pathname; same 5 patterns.

Side-effect bootstrap (MutationObserver for manual-capture bridge)
moved out of the IIFE into an explicit `installManualCaptureBridge(doc?)`
so callers invoke it deliberately. Phase 3b content-script entrypoint
files call this at startup; utility imports don't.

## D. `interceptor-ai-patterns.ts` refactor

Legacy and Phase 2 shape:

```ts
export default defineUnlistedScript(() => {
  const AI_API_DOMAINS = new Set([...]);
  function matchUrl(url) { ... }
  // ...
  window.__PCE_AI_PATTERNS = { matchUrl, ... };
});
```

Phase 3a shape:

```ts
export const AI_API_DOMAINS = new Set([...]);
export function matchUrl(url: string): UrlMatch { ... }
// ... all tables + matchers at module level ...

export default defineUnlistedScript(() => {
  if (typeof window !== "undefined") {
    window.__PCE_AI_PATTERNS = { matchUrl, ... };
  }
});
```

Runtime effect: zero change. The unlisted-script bundle still loads
into the page context and still publishes the same API on the same
window global in the same order. All that moves is which lexical
scope the symbols live in ‚Äî and ESM exports can travel with them
without any runtime overhead.

Why: the pattern tables + matchers are the single highest-leverage
bit of pure logic in the whole extension (they govern what we
capture vs what we drop). Unit coverage on them is worth more than
on any DOM helper, but the closure layout made them untestable.

## E. Vitest harness shape

### `vitest.config.ts`

- Environment: `happy-dom` by default. Lighter than `jsdom`; ships
  everything we need (`Document`, `Window`, `URL`, `MutationObserver`).
- Tests that don't need DOM opt into node via `// @vitest-environment
  node` at the top of the file (used by `site-configs.test.ts`,
  `injector.test.ts`, `interceptor-ai-patterns.test.ts`,
  `background.test.ts`).
- Coverage: v8 provider, reporter `text + html + lcov`. Scoped to
  `utils/**` + `entrypoints/**`, excluding test files + `.d.ts`.
- Setup file: `test/setup.ts`.
- Alias: `@` ‚Üí repo root for ergonomic imports (matches tsconfig).

### `test/setup.ts`

Two concerns:

1. Mute `console.debug` (selector-engine cache invalidation chatter,
   DOM extraction debug logs ‚Äî they'd drown out real test output).
2. Install no-op stubs for `defineBackground`,
   `defineContentScript`, `defineUnlistedScript` globals. WXT injects
   these at build time; under Vitest they don't exist. Tests that
   import `background.ts` / `interceptor-ai-patterns.ts` need them.
   The stub just returns the argument unchanged ‚Äî harmless, since
   `__testing` hooks live at module scope outside those closures.

### Fake IndexedDB

Imported only by `capture-queue.test.ts` via
`import "fake-indexeddb/auto"` at the top. Happy-dom does not ship
IDB; the `/auto` entry registers the shim on global `indexedDB`.
Keeping the import file-local instead of global-in-setup prevents
leakage ‚Äî tests that don't need IDB stay clean.

## F. Test coverage highlights (149 cases)

Full list in the README. High-signal wins:

- **`interceptor-ai-patterns`** ‚Äî 34 cases locking down every confidence
  path (`high` / `medium` / `none` / `aggressive` variants). Web UI
  noise filtering (sentinel, accounts, settings) + AI-specific path
  acceptance (backend-api/conversation, append-message) + unknown-host
  path-pattern fallback. Every combo of URL-match ‚àß body-match
  promotion.
- **`capture-queue`** ‚Äî 14 cases against `fake-indexeddb`. TTL
  eviction uses a clever trick: stub `Date.now` to the past, enqueue
  a stale record, restore `Date.now`, enqueue a fresh record, flush
  once, assert only the fresh one hit the network. The
  concurrency-guard test races two `flush()` calls and asserts the
  second short-circuits.
- **`selector-engine` cache** ‚Äî 3-strikes-invalidate test removes
  the matching element from the DOM, runs 3 queries, asserts the
  cache entry is cleared + the logger was called.
- **`background.__testing.isDuplicate`** ‚Äî uses `vi.useFakeTimers()`
  + `setSystemTime` to fast-forward past the 5-second dedup window
  and assert the second call fires fresh.

## G. Import-time caveat for `background.ts`

`background.ts` calls `defineBackground(() => { ... })` at module
scope. `test/setup.ts` stubs `defineBackground` to a pass-through
no-op so the import doesn't throw. The `main()` body (which touches
`chrome.runtime.onMessage` etc.) is never executed under Vitest
because the stub never invokes its argument.

This is the minimum-invasive approach: it doesn't require factoring
`background.ts` into "pure module + runtime shell" files. The cost
is one added line of config per entrypoint-importing test
(`// @vitest-environment node` because we don't need a DOM for those
tests). Worth it.

## H. Known limits (accepted)

1. **Stale IDE lints pre-install.** `vitest`, `vitest/config`,
   `node:path`, `node:url` all resolve post-`pnpm install`. README
   documents this explicitly so reviewers aren't surprised. Same
   class as the `wxt`, `@types/chrome` pre-install lints we already
   have.
2. **`background.ts` message routing is not unit-tested.**
   `handleCapture` / `handleNetworkCapture` / `handleSnippet` all
   live inside the `defineBackground` closure and call into
   `chrome.runtime` + `fetch` + `CaptureQueue`. Factoring them into
   pure helpers for testability would be a behaviour-changing
   refactor larger than this PR. They land in a Phase 3b+
   chrome-messaging integration harness.
3. **Site-config selectors are data, not logic.** The tests assert
   every provider has a config entry + every alias is object-identical
   to its target, but we don't exercise every selector against live
   DOM fixtures. Phase 3b ports each site extractor with its own
   DOM-fixture tests.
4. **`happy-dom` vs real-browser semantics drift.** Possible edge
   cases where happy-dom's `innerText` / `MutationObserver` / `URL`
   implementations diverge from real Chrome. For now we accept this
   as a known limit and rely on the existing Python-Selenium e2e
   suite (landing on the WXT bundle in Phase 4) to catch real-browser
   regressions.
