## A. What this PR does

Phase 2 of the 4-phase WXT/TypeScript migration scoped in ADR-006. Ports
the content-script bridge + all three page-context interceptor scripts
to TypeScript. Pure 1-for-1 behaviour port — every regex, every noise
filter, every WS-protocol quirk, every `toString` disguise is preserved.

## B. Why the file shape changed

The legacy layout was:

```
pce_browser_extension/
├── content_scripts/bridge.js          # runs in ISOLATED world (content script)
└── interceptor/
    ├── page_confirmed.js              # injected into MAIN world via <script src>
    ├── ai_patterns.js                 # "
    └── network_interceptor.js         # "
```

For WXT, the natural TS layout is:

```
pce_browser_extension_wxt/entrypoints/
├── bridge.content.ts                  # WXT auto-registers as content script
├── interceptor-page-confirmed.ts      # WXT unlisted script → output root
├── interceptor-ai-patterns.ts         # "
└── interceptor-network.ts             # "
```

WXT's naming convention for unlisted scripts is `entrypoints/<name>.ts`
without a `.content` / `.background` suffix, which emits
`<output>/<name>.js`. Subdirectories like `entrypoints/interceptor/`
are not a documented unlisted-script convention as of WXT 0.19, so
we flattened to `interceptor-<name>.ts`. `bridge.content.ts` registers
itself via `defineContentScript` and no longer needs to sit in the
imperative `content_scripts` list in `wxt.config.ts`.

## D. Build-chain implications

- **New manifest entries.** WXT auto-registers `bridge.content.ts` as a
  content script with the 17 match patterns declared inline. It also
  emits `interceptor-page-confirmed.js` / `interceptor-ai-patterns.js`
  / `interceptor-network.js` at the output root and adds them to
  `web_accessible_resources` (we made that explicit in `wxt.config.ts`).
  The emitted manifest now looks like:
  ```jsonc
  {
    "content_scripts": [
      { /* 17-site bridge from bridge.content.ts (auto) */ },
      /* …15 site bundles (legacy JS, unchanged) */
      /* …1 universal detector bundle (legacy JS) */
    ],
    "web_accessible_resources": [
      { "resources": ["interceptor-page-confirmed.js",
                      "interceptor-ai-patterns.js",
                      "interceptor-network.js"], "matches": ["<all_urls>"] }
    ]
  }
  ```
- **No new runtime deps.** Zero change to `package.json`'s
  `dependencies` / `devDependencies`. Vitest + `fake-indexeddb` land in
  Phase 3a as part of the test-harness PR.
- **No change to public/ after sync.** `public/` now contains only
  `content_scripts/`, `icons/`, `popup/`. The legacy `interceptor/`
  directory is no longer staged.

## E. Type-shim deltas

`types.d.ts` grew two global-scope function signatures
(`defineContentScript`, `defineUnlistedScript`) because those are
referenced by the four new entrypoint files. We also moved the
`chrome` namespace inside `declare global { ... }` after catching a
subtle TypeScript bug: putting `export {}` at the bottom of the file
turns the whole file into a module, which scopes any top-level
`declare namespace chrome` _out_ of the global scope. The fix is
to declare everything inside `declare global { ... }` and keep the
trailing `export {};`. Pre-install `tsc --noEmit` is clean again.

Post-install the real `@types/chrome` + WXT's generated
`.wxt/types/*.d.ts` supersede these stubs via TS declaration merging.

## F. Known limits (accepted)

1. **`simpleHash` in `network_interceptor.js` was dead code** — defined
   but never called in the legacy JS. We dropped it from the TS port
   rather than port unused code. If future needs arise it lives
   identically in `background.ts.__testing.simpleHash`.
2. **`_bodyInit` Request-clone path is non-standard.** The legacy JS
   read `Request.prototype._bodyInit` as a best-effort Svelte/React
   compat hack. We preserved this via an `as any` cast; it will be
   `undefined` on modern browsers, which reproduces the legacy's
   fall-through behaviour (no capture of POST body for wrapped
   `Request` objects).
3. **WS protocol extractor uses lots of `as any`.** The ChatGPT /
   Claude WS message shapes are private and change frequently. Typing
   them strictly would be brittle; the legacy JS is similarly loose.
   We kept runtime checks identical and used `Record<string, unknown>`
   + localized `any` casts for the deep-nested access.
4. **Bridge matches duplicated between `bridge.content.ts` and
   `wxt.config.ts`.** The 17 site matches appear in both the bridge's
   `defineContentScript` call and the 15 imperative site bundles in
   `wxt.config.ts`. This is **by design** for Phase 2: the site
   bundles must keep matching until Phase 3 ports the site extractors.
   Once Phase 3 lands, each site extractor becomes its own
   `entrypoints/<site>.content.ts` with its own match list, and the
   bridge remains as the single shared content-script bridge.
