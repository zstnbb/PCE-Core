## Summary

The Phase 4 capstone. Ran a clean `pnpm install && pnpm build` against
`pce_browser_extension_wxt/`, produced a 3.7 MB MV3 bundle (14 site
content scripts + interceptors + detector + behaviour tracker), then
drove a full 14-site real-browser regression via
`tests/e2e/pce_site_probe.py` against a managed Chrome 147 profile.

**Verdict**: the WXT-migrated extension ships. **8 sites pass the full
three-phase probe end-to-end** on the fresh build (connectivity →
capture + session + roles → dashboard render). The remaining 6 sites
split into **3 login-wall skips** (orthogonal) and **3 pre-existing
extractor-pipeline gaps** that the state file confirms were failing
identically *before* the P2.5 migration — so no P2.5 phase introduces
a regression.

One adapter-side fix shipped alongside
(`tests/e2e/sites/copilot.py` provider-name alignment); the three
extractor gaps are documented as follow-up tickets, not in-scope for a
validation phase.

### 4. Root-cause: one adapter-side bug, not an extension regression

Comparing extractor output vs e2e adapter expectations turned up a
single genuine test-harness defect:

- `pce_browser_extension_wxt/entrypoints/copilot.content.ts:150`
  emits `provider: "microsoft"` (OpenInference-aligned vendor naming).
- `tests/e2e/sites/copilot.py:15` declared `provider = "copilot"`.

So the probe's `wait_for_new_captures(provider="copilot")` and
`wait_for_session_with_messages(provider="copilot")` never saw the
captures that were actually landing under `microsoft`. Fixed in one
edit:

```diff
 class CopilotAdapter(BaseSiteAdapter):
     name = "copilot"
-    provider = "copilot"
+    # The WXT extractor (entrypoints/copilot.content.ts) emits
+    # provider="microsoft" to align with OpenInference vendor naming.
+    # Keep the adapter in lockstep so wait_for_new_captures() and
+    # wait_for_session_with_messages() actually see copilot traffic.
+    provider = "microsoft"
     url = "https://copilot.microsoft.com/"
```

All 14 adapters were cross-checked against their extractor counterparts
(grep over `provider:` in `.content.ts` vs `provider =` in
`tests/e2e/sites/*.py`). Every other pair is aligned. Mapping

### 7. Root-cause #2: DOM extractors emit only `network_intercept`, never `conversation`

The copilot and huggingface P2 failures both present the identical
symptom `no user+assistant session (roles seen: [])`. Peeking into
the DB immediately after the rerun (via the live `/api/v1/*`
endpoints):

```
stats.by_provider: poe=47, anthropic=12, moonshot=11, microsoft=10,
                   xai=10, openai=4, deepseek=3, google=2,
                   huggingface=2, zhipu=2, manus=1, unknown=61
stats.by_direction: conversation=12, network_intercept=153
sessions_count=11
```

Captures are arriving for microsoft (10), huggingface (2), and openai
(4) — the provider classifier **is** working. But **zero sessions
exist for those three providers**:

- microsoft: 10 captures · 0 sessions
- huggingface: 2 captures · 0 sessions
- openai: 4 captures · 0 sessions

Meanwhile, each of the 8 passing sites produces its session with the
expected 2–3 messages. Cross-referencing with `direction`:

- **Passing sites** produce `direction=conversation` captures from
  the per-site DOM extractor (e.g. `deepseek.content.ts`,
  `manus.content.ts`), which encode turn-structured
  `{role, content, attachments}` payloads the normalizer can promote
  into sessions.
- **Failing sites** (copilot / huggingface / chatgpt) produce **only
  `direction=network_intercept` captures** — raw fetch/XHR blobs from
  `interceptor-network.js`. The normalizer's current pipeline
  (`pce_core/normalizer/pipeline.py` + `reconciler.py`) does not yet
  synthesise turn structure from network blobs alone, so no session
  gets created.

Put differently: the copilot / huggingface / chatgpt **DOM extractors
are silently failing** on the current site DOMs — they load (manifest
matches, content script executes), but their DOM-walk returns zero
turns, so no conversation-direction capture is emitted. Only the
network interceptor fires.

## Design Decisions

### Why text-only, not all three scenarios?

Phase 4d's scope per ADR-006 is "verify the real build captures end-
to-end across all sites." `text` is the minimum scenario that
exercises the full pipeline (input finding, message send, response
wait, capture arrival, session normalization, dashboard render) for
every site. File and image scenarios additionally exercise native
upload widgets which vary wildly per site; re-running them after the
rebuild is a lower-priority follow-up, and the state file still
preserves the 2026-04-06 file/image baseline for every site that
supported it then (9/14 — deepseek, gemini, googleaistudio, grok,
kimi, manus, perplexity, poe, zhipu all had P2 + P3 green for both
attachment types; that history stands untouched).

Running all 36 (site × scenario) combinations unattended in this
session would have taken ~40 minutes vs ~7 minutes for text-only, and
would not have changed the "did the rebuild regress anything" signal
beyond what the text run already tells us.

### Why fix only the copilot adapter, not chatgpt / huggingface too?

- **copilot**: a 1-line test-harness alignment to the extractor
  contract. Zero risk, obviously correct, immediately makes the
  remaining failure legible (shifts the error from "no captures at
  all" to "captures arrive but have no DOM turns"). Ships.
- **chatgpt**: `input not found after navigate` is an adapter-flow
  issue — after P1 finds the input, the P2 `navigate()` re-loads the
  page which surfaces ChatGPT's project-picker modal, hiding the
  composer. Fixable but requires DOM reconnaissance + possibly a
  "dismiss modal" helper. Out of scope for a validation phase.
- **huggingface**: the 60-s response timeout + zero conversation
  captures strongly suggests the HF Chat DOM structure has diverged
  from what `huggingface.content.ts` expects (selectors from batch 4,
  2026-04-18 morning, haven't been re-tuned since the HF UI update).
  Needs live DOM inspection + selector audit. Out of scope.

Silently fixing two unrelated extractors inside a validation note
would muddle the record. They get their own tickets (below) so the
fix is reviewable in isolation.

### Why not run this under Edge (which doesn't have the Chrome 137+ blocker)?

The probe driver supports `--browser edge`, but the Edge managed
profile (`~/.pce/edge_profile/`) has no logins — running under Edge
would force manual login on 14 sites, which defeats the goal of a
repeatable unattended regression. The managed Chrome profile already
has the extension pre-installed (Chrome 137+ accepts pre-registered
unpacked extensions even though it rejects `--load-extension` on
cmdline), so Chrome is strictly faster.

### Why the "page PCE flags: <none>" diagnostic fires for every site?

The probe logs `page PCE flags: <none>` across every site, including
the 8 sites that pass. This is a **false-negative diagnostic** baked
into `_probe_extension_state()` in `pce_site_probe.py`: it reads
`window.__PCE_*_ACTIVE` via `driver.execute_script(...)`, which runs
in Chrome's **MAIN world**. MV3 content scripts run in the **isolated
world** by default (including WXT's output), so MAIN-world JS cannot
see those globals.

The diagnostic was written against the pre-WXT extension which
injected flags into MAIN world via `<script>` tags. It's now
permanently stale. Two options:

1. Delete the diagnostic outright (noisy, misleading).
2. Rewire it to query the content script via `chrome.runtime.sendMessage`
   through the existing `bridge.content.ts` MAIN-world bridge.

Deferred to a separate tiny PR; it's confusing but not a blocker — the
real signal (P2 pass/fail + capture count + session roles) is accurate.

## Risks mitigated

- **"Does the rebuilt bundle actually run on real sites?"** — yes.
  8 sites produced fresh sessions with fresh IDs immediately after the
  rebuild, proving the WXT output is being loaded by Chrome 147 and
  emitting capture events into the pipeline.
- **"Did any phase of P2.5 break a site that worked before?"** — no.
  Every failing site today was failing identically in the
  pre-rebuild state file (timestamp `1776477xxx`, 2026-04-17). The
  extractors that passed before pass now.
- **Adapter-extractor contract drift** — closed for copilot. Every
  other pair is aligned; the grep is cheap to re-run in future
  phases (`grep "provider:" entrypoints/*.content.ts` vs
  `grep "provider =" tests/e2e/sites/*.py`).

## Remaining risk

- Three site-specific DOM extractors (chatgpt / copilot / huggingface)
  are silently emitting zero conversation captures today. They still
  produce network-intercept traffic, so the pipeline isn't *broken* —
  it just only sees the raw API layer. Follow-up tickets cover
  re-tuning the DOM walks OR teaching the normalizer to build
  sessions from network-intercept payloads.
- Three site sessions (claude / mistral / perplexity) await a user
  login refresh to close coverage. The extension code is blameless
  there.
