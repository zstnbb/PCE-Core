## Summary

Final batch of the site-extractor migration. Ports the last four
legacy content scripts + the dynamically-injected universal extractor
to TypeScript:

1. `generic.js` â†’ `entrypoints/generic.content.ts` (Mistral + Kimi).
2. `detector.js` â†’ `entrypoints/detector.content.ts` (`<all_urls>` except covered).
3. `behavior_tracker.js` â†’ `entrypoints/behavior-tracker.content.ts`.
4. `text_collector.js` â†’ `entrypoints/text-collector.content.ts`.
5. `universal_extractor.js` â†’ `entrypoints/universal-extractor.ts` (unlisted; injected dynamically by the service worker).

**After this batch, Phase 3b is complete.** Every PCE content-script-
side module is TypeScript. The imperative `content_scripts` list in
`wxt.config.ts` is now empty â€” WXT auto-registers all 18 entrypoints
(1 background + 3 interceptors + 1 bridge + 13 site extractors +
detector + behavior-tracker + text-collector, plus 1 dynamically-
injected unlisted universal extractor). The legacy JS files under
`../pce_browser_extension/content_scripts/` are no longer referenced
by the manifest. Phase 4 removes them + the sync script.

**Matches**: `chat.mistral.ai`, `kimi.moonshot.cn`, `www.kimi.com`,

### 2. `entrypoints/detector.content.ts` (~270 lines)

**Matches**: `<all_urls>` excluding the 13 covered sites.

Scored AI-page heuristic. Fast-path check against 34 known AI
domains; otherwise combines URL path prefixes, title keywords,
meta-description keywords, and 25 DOM signals with per-selector
weights. Posts `PCE_AI_PAGE_DETECTED` to the background worker when
score â‰Ą 6 (`HEURISTIC_THRESHOLD`).

All tables (`KNOWN_AI_DOMAINS`, `KNOWN_AI_PATH_PREFIXES`, `DOM_SIGNALS`,
`TITLE_KEYWORDS`, `META_KEYWORDS`) and the `detectAIPage` function are
exported at module scope so the Vitest suite can exercise the
detector without running the full entrypoint closure.

Run at `document_idle` (same as legacy) â€” re-scans every 3 s for SPA
URL changes.

### 3. `entrypoints/behavior-tracker.content.ts` (~300 lines)

**Matches**: All 17 covered sites (13 TS extractor sites + the Mistral + 3
Kimi hosts covered by `generic.content.ts`).

Writes `window.__PCE_BEHAVIOR.getBehaviorSnapshot(reset)` consumed by
the capture runtime. Tracks:

  - **`lastRequestSentAt`**: timestamp of last send-button click OR
    Enter keypress in a textarea / contenteditable.
  - **`copyEvents`**: last 10 copy events with timestamp + selected
    text length.
  - **`scrollDepthPct`**: max scroll depth (0-100) reached.
  - **`messageTimestamps`**: append-only log; `think_interval_ms` =
    last - second-to-last when â‰Ą 2 are present.

Pure helpers exposed at module scope:
  - `createBehaviorState()` â†’ fresh zero-valued state object.
  - `getBehaviorSnapshot(state, reset)` â†’ snapshot + optional reset
    of transient state (copy events + scroll depth).
  - `isSendButtonTarget(el)` â†’ classifies a click target against the 6
    known send-button selectors + the generic "button inside a form
    with a textarea" fallback.
  - `isSendKeyEvent({key, shiftKey, target})` â†’ classifies a keydown
    event (Enter without Shift in a textarea / contenteditable).
  - `computeScrollDepth(scrollTop, scrollHeight, clientHeight)` â†’
    integer 0-100.

### 4. `entrypoints/text-collector.content.ts` (~340 lines)

**Matches**: All 17 covered sites.

Byte-for-byte port of the legacy floating "Save Snippet" button.
Inline `<style>` block + 5-category picker (general / code / prompt /
output / reference). Emits `PCE_SNIPPET` messages to the background
worker, which POSTs to `/api/v1/snippets`.

Pure helpers at module scope:
  - `DOMAIN_PROVIDER_MAP` â†’ complete 17-site map.
  - `CATEGORIES` â†’ 5 category defs with ids + labels + icons.
  - `detectProvider(hostname)` â†’ map hit, else second-to-last
    hostname label.
  - `clampFloatingPosition(x, y, vw, vh)` â†’ viewport-clamped
    floating-button coordinates.
  - `MIN_SELECTION_LENGTH = 5` â€” same threshold as legacy.

Runtime at `document_idle` (UI helper; doesn't need early injection).

### 5. `entrypoints/universal-extractor.ts` (~360 lines)

**Registered as**: `defineUnlistedScript`. WXT emits
`universal-extractor.js` at the build output root. The background
worker dynamically injects it via `chrome.scripting.executeScript`
when the detector flags an AI page on an unknown domain.

Differences from legacy `universal_extractor.js`:

  - **Uses the shared capture runtime** (`utils/capture-runtime.ts`)
    rather than re-implementing the observer + fingerprint + debounce
    machinery. Result: ~360 TS lines vs. 460 legacy lines, with
    behaviour-parity fingerprint + rollback semantics guaranteed by
    the existing runtime tests.
  - **Imports `utils/pce-dom.ts` helpers directly** (extractAttachments,
    extractReplyContent, extractThinking, fingerprintConversation,
    getSessionHint, isStreaming) instead of reading them from
    `window.__PCE_EXTRACT`. The dynamically-injected script ships
    those helpers inlined via ESM bundling.
  - **Site-specific flag guard preserved** â€” bails out fast if any of
    the 13 `__PCE_*_ACTIVE` flags is set, so the universal extractor
    never double-captures on a site that already has a dedicated TS
    extractor.
  - **Container-detection algorithm preserved** â€” the scored
    `CONTAINER_SELECTORS` list, the `childCount * 0.5` adjustment,
    and the 13 MESSAGE_SELECTORS + 8 USER / 9 ASSISTANT role
    predicates are lifted byte-for-byte.
  - **Provider guessing** uses `window.__PCE_DETECTOR.KNOWN_AI_DOMAINS`

(replaces the legacy `__PCE_UNIVERSAL_*` family of debug globals).

## Design Decisions

### Why promote `universal-extractor` to the shared capture runtime?

The legacy `universal_extractor.js` re-implemented ~200 lines of
observer + debounce + fingerprint + rollback logic that the capture
runtime already covers. Re-using the runtime:

  1. Eliminates duplicate code.
  2. Inherits the `requireBothRoles`, `isStreaming`, and rollback
     behaviours that were added / tested in batches 1-4.
  3. Reduces the universal extractor to ~360 lines (vs. 460 legacy)
     with tighter coverage of the extraction-specific logic.

No behaviour change is expected â€” the legacy observer + rollback
semantics are preserved exactly by the shared runtime. Any drift
would show up as a capture-runtime regression caught by the 16+
runtime tests.

### Why doesn't `injector.ts` switch to `universal-extractor.js` in this PR?

`getDynamicScriptsForDomain` still reads from `DOMAIN_CONTENT_SCRIPTS`
when the domain matches a known-AI-site entry, and those entries
reference legacy filenames like `content_scripts/chatgpt.js`. Switching
the fallback path independently would produce an inconsistent mix. 
Phase 4 will:

  1. Delete `DOMAIN_CONTENT_SCRIPTS` (no longer needed now that all
     extractors are WXT defineContentScript entries that auto-register
     on their matches).
  2. Simplify `getDynamicScriptsForDomain` to return
     `[UNIVERSAL_EXTRACTOR_TS_PATH, "content-scripts/behavior-tracker.js"]`
     (or however WXT names those outputs).
  3. Delete the legacy `content_scripts/*.js` sync step.

### Why is the detector `excludeMatches` hardcoded in `detector.content.ts`?

WXT's `defineContentScript` emits an MV3 manifest entry with the
`matches` + `exclude_matches` we declare. The list must be static
(it's in the manifest). We could lift the 13-site list to
`wxt.config.ts` and import it into `detector.content.ts`, but WXT
has limited support for importing build-time constants into
content-script entrypoints. Duplicating the list is the pragmatic
choice; Phase 4 can factor it once if needed.

### Why keep `sync-legacy-assets.mjs` alive?

Two reasons:
  1. `injector.ts`'s `DOMAIN_CONTENT_SCRIPTS` + `UNIVERSAL_FALLBACK_SCRIPTS`
     still reference `content_scripts/*.js` filenames (for the
     dynamic-injection happy path on known AI sites). Removing the
     sync would break those calls.
  2. Preserving the legacy dir until Phase 4 keeps the migration
     reversible: if e2e testing reveals a regression in any TS
     extractor, the legacy JS is still trivially re-wirable.

Phase 4 handles the cleanup as one atomic sweep.

### Why ship `universal-extractor.ts` as `defineUnlistedScript` instead of `defineContentScript` with a dynamic-register path?

`chrome.scripting.executeScript` expects a file path relative to the
extension root. `defineUnlistedScript` emits the script there. The
alternative is to `chrome.scripting.registerContentScripts` with a
declarative config, but that mutates the manifest dynamically and
is harder to reason about. The legacy code used
`chrome.scripting.executeScript` with a file path, and we preserve
that approach.

### Why doesn't the runtime register the behavior tracker itself?

Keeping the behavior tracker as a separate content script means it
runs **before** the site extractor (MV3 ordering guarantee given
matching `run_at`) and is available when the extractor's first
capture fires. The capture runtime reads `window.__PCE_BEHAVIOR?`
lazily â€” so even if the order flips, the `{}` fallback ensures
nothing crashes.

## Risks mitigated

- **Dynamic-injection fallback regressions**: `UNIVERSAL_EXTRACTOR_TS_PATH`
  constant lands now; `injector.ts`'s runtime path stays on the legacy
  filename until Phase 4 wires things over. Zero risk in this PR.
- **Manifest drift**: `content_scripts: []` + empty `legacy*` constants
  means no accidental legacy-JS references remain in the manifest.
  `wxt.config.ts` self-documents the migration complete.
- **Universal-extractor dedup**: `SITE_EXTRACTOR_FLAGS` guard +
  capture-runtime's fingerprint-diff guarantee the universal extractor
  never double-captures alongside a site-specific one.
- **Behavior snapshot drift**: `getBehaviorSnapshot`'s pure-function
  signature is test-covered; the entrypoint `main()` closure remains
  a thin wiring layer over tested primitives.
