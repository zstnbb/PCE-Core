# P2.5 Phase 1 — Background Service-Worker TypeScript Port

- Date: 2026-04-17
- Phase: Industrialization P2.5 — Browser Extension TS Migration, Phase 1 of 4
- Follows: `2026-04-17-P2-completion.md`
- Scope: Legacy `background/` JavaScript → TypeScript under WXT
- Guardrail: ADR-006 §"迁移过程中不引入任何新功能，保持'结构替换、行为不变'"

## A. Plan

P2.5 splits the 175 KB of legacy extension code into four reviewable
phases. Phase 1 — this PR — covers the background service worker only.
It is an **exact** 1-for-1 port: every handler, timer, dedup window,
state transition and fetch shape is preserved. Nothing changes for the
content scripts, the popup, or the e2e tests.

The four phases:

| Phase | Scope | Status |
|---|---|---|
| 1 | `background/*.js` → TS under `entrypoints/` | ✅ this PR |
| 2 | `bridge.js` + `network_interceptor.js` + shared helpers + Vitest | ⏳ next PR |
| 3 | 13 site extractors, 2-3 per PR (5 PRs total) | ⏳ |
| 4 | Delete legacy dir + sync script + update root docs | ⏳ |

Each phase is independently reverable by dropping its PR(s) — the legacy
extension under `../pce_browser_extension/` remains untouched and
continues to ship verbatim until Phase 4.

### Rewritten
- `pce_browser_extension_wxt/entrypoints/background.ts` — **625
  lines of TypeScript**. Replaces the 40-line thin wrapper that
  previously delegated to `entrypoints/legacy/service_worker.js`.
  Full list of ported subsystems:
  - Message router (7 types: `PCE_CAPTURE`, `PCE_NETWORK_CAPTURE`,
    `PCE_SNIPPET`, `PCE_GET_STATUS`, `PCE_SET_ENABLED`,
    `PCE_BLACKLIST_UPDATED`, `PCE_AI_PAGE_DETECTED`).
  - Capture handlers (`handleCapture`, `handleNetworkCapture`,
    `handleSnippet`) — typed `IngestBody` builder, bit-identical
    field mapping.
  - Dedup — 5 s short-time window keyed by `session_hint +
    simpleHash(fingerprint)`, periodic cleanup.
  - Self-check — per-domain detection vs capture counter, badges
    the icon with `!` when an AI page was detected but no capture
    arrived within 30 s.
  - Context menu — "Capture This Page (PCE)" manual capture.
  - Tab lifecycle — `injectedTabs` set maintained across
    `tabs.onRemoved` / full navigation; `proactiveInjectFallback`
    fires after `status === "complete"`.
  - Health loop — polls `/api/v1/health` every 30 s, exposes via
    `PCE_GET_STATUS`.
  - Offline queue — routes captures through `CaptureQueue.enqueue`
    on 5xx / timeout, starts retry loop at boot.
  - Discovered-domain reporting — fire-and-forget to
    `/api/v1/domains`, per-session dedup via `reportedDomains`.
  - Persisted state (`chrome.storage.local`) — `pce_enabled`,
    `pce_capture_count`, `pce_blacklist`.
  - `__testing` hook exposing `simpleHash` / `isDuplicate` /
    `isBlacklisted` / `recordDetection` / `recordCaptureSuccess` /
    `getSilentDomains` plus `resetState` / `getState` so Vitest can
    exercise every pure helper without a real chrome runtime.

### Module boundaries that make Phase 2 cheap

- `injector.ts` exposes pure helpers with no `chrome.*` dependency:
  `matchDomainConfig`, `toContentScriptPaths`,
  `getExtractorFlagForDomain`, `getDynamicScriptsForDomain`,
  `getScriptsForUrl`. These are the first unit-test targets once the
  Vitest harness lands.
- `background.ts` exports a `__testing` object with the pure helpers
  (`simpleHash`, `isDuplicate`, `isBlacklisted`) plus state accessors
  (`getState`, `resetState`). Contract is stable enough for Phase 2 to

- Every fetch URL is resolved via
  `DEFAULT_PCE_SERVER_ORIGIN` / `DEFAULT_PCE_INGEST_URL` from
  `utils/pce-messages.ts`, so a future runtime-config PR can flip the
  origin to an env var without touching the capture paths.

## D. What Does Not Work Yet / Known Limits

1. **`pnpm install` + `pnpm typecheck` not run in this session.** The
   skeleton is structurally complete and follows the WXT 0.19 schema,
   but actually executing `pnpm` requires the developer's Node
   workspace. The pre-install Chrome stub in `types.d.ts` means a
   fresh clone doesn't show phantom "cannot find name 'chrome'"
   errors; once `pnpm install` pulls in `@types/chrome`, the real
   types supersede the stub.
2. **No Vitest harness yet.** Unit tests for `injector.ts` pure
   helpers, `capture-queue.ts` with `fake-indexeddb`, and the
   `background.ts` `__testing` hooks are the headline deliverable for
   Phase 2. The `__testing` exports are deliberately placeholder-free
   so those tests can land without touching the main file.
3. **Site extractors still legacy JS.** All 13 per-site content
   scripts (ChatGPT, Claude, Gemini, …) continue to ride through as
   `.js` under `public/content_scripts/`. Phase 3 rewrites them 2-3
   per PR in traffic-weight order.
4. **`bridge.js` + `network_interceptor.js` still legacy JS.** These
   are the shared plumbing consumed by every site extractor; Phase 2
   ports them before Phase 3 touches a single extractor.
5. **Popup UI untouched.** `popup/popup.{html,js,css}` remain
   verbatim under `public/popup/`. The popup currently talks to the
   background via `chrome.runtime.sendMessage({type: "PCE_GET_STATUS"
   | "PCE_SET_ENABLED" | "PCE_BLACKLIST_UPDATED"})`, all of which the
   new background.ts handles identically. No popup changes required.
6. **Coarse chrome stub in `types.d.ts`.** Pre-install the stub is
   `any`-typed. That means a typo in a `chrome.*` call won't be
   caught until after `pnpm install`. Trade-off: it keeps the
   skeleton type-check-clean from a fresh clone, which matters more
   than exhaustive pre-install type coverage.

### WXT typecheck + build (requires Node + pnpm locally)

```bash
cd pce_browser_extension_wxt
pnpm install                    # pulls wxt, typescript, @types/chrome
pnpm typecheck                  # tsc --noEmit; should be clean
pnpm build                      # Chrome bundle at .output/chrome-mv3/
pnpm build:firefox              # Firefox bundle at .output/firefox-mv3/
```

### Verify the TS background at runtime

1. `pnpm build` — produces `.output/chrome-mv3/background.js` (bundled
   TS output, not a re-exported JS shim).
2. `chrome://extensions` → Developer mode → Load unpacked →
   `.output/chrome-mv3/`.
3. Open any covered AI site. Expect identical capture rows in the
   dashboard as before the port. The dashboard query that proves it:
   ```sql
   SELECT source_type, source_name, direction, host, COUNT(*)
   FROM captures
   WHERE source_type = 'browser_extension'
   GROUP BY source_name, direction, host;
   ```
   Pre- and post-port row shapes should match exactly.

## F. Risks / Follow-ups

### Risks surfaced during Phase 1

- **`chrome.runtime.onMessage` return-value semantics are subtle.**
  MV3 treats `true` as "async response, don't auto-close the port".
  The legacy JS relied on this convention; the TS port preserves it
  verbatim (`return true` for async handlers, `return false` for
  sync). Any future refactor that changes return types must keep this
  property — otherwise popups/content scripts will hit "port closed"
  errors.
- **`AbortSignal.timeout` is MV3-only (Chromium ≥ 103).** The legacy
  code uses it; the TS port preserves it. Our manifest requires
  `strict_min_version: "109.0"` on Firefox and MV3 on Chrome, both
  of which support it. No change.
- **`indexedDB` lives in the service-worker global scope under MV3.**
  The port assumes the same, as did the legacy code. This is safe
  for now. If Chrome ever ships a SW-restart that tears down IDB
  between messages (it currently preserves IDB across restarts), the
  queue could double-enqueue — same risk as legacy.

### Recommended Phase 2 follow-ups

1. **Vitest harness.** Add `vitest` + `@vitest/ui` + `fake-indexeddb`
   to `devDependencies`. Test targets: `injector.ts` pure helpers,
   `capture-queue.ts` IDB semantics, `background.ts` `__testing`
   hooks.
2. **`bridge.js` → `entrypoints/bridge.content.ts`.** Small file
   (~120 lines); ports cleanly.
3. **`network_interceptor.js` → `entrypoints/interceptor/*.ts`.**
   Larger (~800 lines); split into `network-interceptor.ts` +
   `ai-patterns.ts` + `page-confirmed.ts`.
4. **Shared helpers — `pce_dom_utils.js` + `selector_engine.js` +
   `site_configs.js` — move into `entrypoints/shared/`.** All 13
   site extractors will import from this tree starting in Phase 3.
5. **Fix the `sync-legacy-assets.mjs` to no-longer copy
   `background/`.** Already accidentally correct because the script
   only copies the four listed dirs (`content_scripts`, `interceptor`,
   `icons`, `popup`) — but worth a comment reminder.
