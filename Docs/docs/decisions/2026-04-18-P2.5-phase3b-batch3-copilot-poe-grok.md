## Summary

Third round of the site-extractor migration. Ports `copilot.js`,
`poe.js`, and `grok.js` to TypeScript using the capture-runtime
factory. One small runtime-API addition: a new `requireBothRoles`
option lifts the "defer capture if only one role is visible" guard
that Poe and Grok had inline in their legacy JS.

**9 of 13 site extractors now on TypeScript.** Remaining: `huggingface`,
`manus`, `zhipu`, `generic` + `universal_extractor` (plus the
standalone `detector` / `behavior_tracker` / `text_collector`).

### 4. Runtime extension — `requireBothRoles` option

Added to `utils/capture-runtime.ts` `CaptureRuntimeOptions`:

```ts
/**
 * If true, the runtime defers capture (at `streamCheckMs` cadence)
 * until the extracted messages contain BOTH a user role and an
 * assistant role. Used by sites whose DOM briefly reveals only one
 * side mid-render (Poe / Grok pattern). Defaults to false.
 */
requireBothRoles?: boolean;
```

Fires inside `captureNow()` after `extractMessages()` returns but
before the fingerprint diff:

```ts
if (options.requireBothRoles) {
  const hasUser = allMsgs.some((m) => m && m.role === "user");
  const hasAssistant = allMsgs.some((m) => m && m.role === "assistant");
  if (!hasUser || !hasAssistant) {
    schedule(STREAM_CHECK_MS);
    return;
  }
}
```

Verified by 3 new cases in `utils/__tests__/capture-runtime.test.ts`:
user-only defer + eventual proceed once assistant arrives, assistant-
only defer, and both-present pass-through (no regression for callers
who don't set the flag).

## Design Decisions

### Why promote `requireBothRoles` to the runtime instead of handling inline in each extractor?

Both Poe's and Grok's legacy JS had the same pattern:

```js
const hasUser = allMessages.some((m) => m.role === "user");
const hasAssistant = allMessages.some((m) => m.role === "assistant");
if (!hasUser || !hasAssistant) {
  scheduleCapture(1500);
  return;
}
```

Keeping it inline in each TS extractor would duplicate this subtle
stateful logic (reschedule + early-return before fingerprint diff)
across two files. If future extractors need the same guard — and
batch 4/5 likely will for at least `manus` or `huggingface` — we'd
keep copy-pasting. Promoting it to the runtime costs ~8 LOC, adds a
single optional flag, and its test coverage in
`capture-runtime.test.ts` guarantees correctness once and for all.

The flag defaults to `false`, so every prior extractor call-site keeps
its current behaviour. Zero regression risk.

### Why not use a named `deferOnIncompleteMessages` instead of `requireBothRoles`?

The runtime's guard is specifically about requiring BOTH roles — it's
not a generic "incomplete message" check. A site that produces only
user messages (search-only interface, say) would set this to `false`.
The name matches what the guard actually asserts.

### Why don't Poe and Grok hook `history.pushState`?

Legacy parity. Both legacy JS files relied on URL polling only. Grok's
legacy file explicitly `reconnectObserver`s on URL change rather than
patching the history API. We mirror that: `hookHistoryApi: false` +
default `pollIntervalMs: 4000 ms`.

### Why does Copilot use `pollIntervalMs: 5000` while Poe and Grok use `4000`?

Legacy parity with each site's own legacy file. The values differ
because the observed rate of SPA navigation differed. Not worth
normalising during the port.

### Why normalizeText stays local to each extractor?

Each site's noise-regex set is idiosyncratic:
  - **Poe**: `HH:MM` timestamps + compare/share/copy/copied
  - **Grok**: share/auto/copy/copied + timing strings + mode toggles
  - **AI Studio** (from batch 2): UI control lines + meta timestamps + disclaimer + download/copy actions

Promoting a generic `normalizeText` to `utils/pce-dom.ts` would invite
either (a) a kitchen-sink regex bundle that mis-strips legitimate
content on the site it wasn't meant for, or (b) a parametrised
`normalizeText(text, regexBundle)` that's actually harder to use
than inlining. Each extractor stays self-contained.

## Risks mitigated

- **Per-site regressions**: 43 new Vitest cases enumerate the
  extraction / role-detection / noise-stripping heuristics for each
  site.
- **Runtime regressions for pre-existing extractors**: The
  `requireBothRoles` flag defaults to `false`, so batches 1-2 sites
  keep their behaviour byte-for-byte. The three new runtime tests
  exercise the new code path without touching existing paths.
- **Poe/Grok incomplete-capture race**: The legacy 1500 ms
  reschedule semantic is preserved exactly by routing through
  `streamCheckMs`.
